# Plan: Hard Paywall After Guided Tour

Generated by `/plan-ceo-review` + `/plan-eng-review` on 2026-06-03. Branch: master.

## Goal

Reorder the new-user flow from **onboarding → (in-onboarding paywall) → opportunistic tour**
to **onboarding → forced interactive tour → hard paywall (no X) → app**. The hard paywall
starts a 3-day free trial. Non-converters fall to the existing free tier after the trial
("trial gate, then freemium"). Today only ~37% of users who see the soft in-onboarding
paywall even tap the trial CTA (Mixpanel, 90d, test accounts excluded); the hard gate forces
the decision after value is delivered by the tour.

## Decisions locked (CEO + Eng review)

| # | Decision | Choice |
|---|----------|--------|
| C1 | Post-trial state | Trial gate, then freemium (free tier reachable only after trial expires) |
| C2 | Tour length before wall | Force all 14 steps, then the wall |
| C3 | Tour abandonment | Resume at the abandoned step on relaunch |
| C4 | Escape on the wall | No X. Keep Restore + Terms + Privacy. No "maybe later". |
| E1 | Gate location | GoRouter `redirect` — single source of truth, re-derived every launch |
| E2 | What clears the wall | One-time entry **latch** set on trial-start / premium / safety-valve |
| E3 | Offerings-load failure | Temp let-in ("Continue") WITHOUT setting the latch → re-wall when healthy |

## Flow state machine

```
                    ┌──────────────┐
  cold launch ─────▶│ !onboarded   │──▶ /welcome (existing)
                    └──────────────┘
                    ┌──────────────────────────┐
  onboarded &       │ tourCompleted == false   │──▶ /tour  (resume @ saved index)
  !tourCompleted    └──────────────────────────┘
                    ┌───────────────────────────────────────┐
  tourCompleted &   │ paywallGateCleared == false &&         │──▶ /hard-paywall
  !latch & !premium │ !premium                               │     (no X)
                    └───────────────────────────────────────┘
  latch || premium  ─────────────────────────────────────────▶ app (home shell)

  Latch set TRUE when:  trial started  OR  premium (gift/referral/RC)  OR
                        (offerings-fail "Continue"  →  latch stays FALSE, re-wall next launch)
```

## Architecture

### 1. Router gate (`lib/core/router.dart:42`)
- The `redirect` is **synchronous** — it cannot await SharedPreferences. So
  `tourCompleted` and `paywallGateCleared` MUST be hydrated into `AppSessionNotifier`
  at boot, exactly like `hasOnboarded` already is (`router.dart:52`). This is the
  highest-risk implementation detail — get it wrong and the gate flickers or fails open.
- Add routes `/tour` (full-screen, rootNavigatorKey) and reuse the hard paywall via a
  new `/onboarding-paywall` route (distinct from the existing soft `/paywall`).
- Extend the redirect: derive stage from `appSession.{hasOnboarded, tourCompleted,
  paywallGateCleared}` + a synchronously-cached premium flag (PurchaseService already
  caches premium per-user in SharedPreferences; surface it on appSession at boot).

### 2. Tour: overlay → mandatory gate (`lib/features/tour/`)
- Today the tour fires opportunistically after `daily.checkinDone` from
  `app_lifecycle_observer.dart:106` / `progress_screen.dart:125`, and on mid-kill it
  restarts at index 0 (prior plan R3). Change to:
  - Start is driven by the router landing on `/tour` (not the lifecycle observer).
  - `advance()` persists the new index to a user-scoped prefs key on every step.
  - On `/tour` mount, resume from the saved index instead of 0.
  - On `tourCompleted`, set `appSession.tourCompleted = true` and let the redirect
    move the user to the hard paywall (controller does NOT navigate imperatively).
- Keep the `guided_tour_enabled` app_config kill switch (fail-open). Add
  `hard_paywall_after_tour_enabled` app_config flag so the whole reorder is
  reversible without redeploy.

### 3. Paywall hard mode (`lib/features/onboarding/screens/paywall_screen.dart`)
- Replace the `inOnboardingFlow` bool with `enum PaywallMode { onboardingHardGate, standaloneUpsell }`.
- `onboardingHardGate`: no X (`_handleClose`, `_showExitOffer`, `_ExitOfferSheet`,
  `paywall_dismiss_count`, refer-unlock routing all disabled). Keep Restore + Terms + Privacy.
- `standaloneUpsell` (existing `/paywall` from journal-save upsell): unchanged — keeps X,
  exit offer, downsell. So none of that code is dead, just mode-gated.
- On successful trial start / purchase: set the `paywallGateCleared` latch (prefs +
  `appSession`), fire `trial_started`, then route to home.
- Offerings-fail safety valve: after N retries (reuse existing `_loadOfferings` error
  path at `paywall_screen.dart:267`), reveal a "Continue" affordance that routes to home
  WITHOUT setting the latch (so the user is re-walled next launch when offerings load).

### 4. Instrumentation
- **New event `trial_started`** — fire in `PurchaseService.purchaseSubscription` success
  (or `_handlePurchase` success). Today NO trial/purchase event reaches Mixpanel; this is
  the only way to measure whether the reorder worked past the CTA tap.
- Ensure `paywall_viewed` fires in the new post-tour placement.
- Tour step drop-off (`tour_step_viewed` / `tour_step_advanced`) already exists — add a
  Mixpanel funnel (onboarding_completed → tour_started → tour_completed → paywall_viewed →
  trial_started) to `docs/analytics/`, excluding test accounts.
- Add a `paywall_offerings_load_failed` event so safety-valve frequency is observable.

### 5. Persistence / migration
- Local prefs (user-scoped via `supabaseSyncService.scopedKey`): `tour_step_index`,
  `paywall_gate_cleared`.
- Migration: add `tour_step_index int` + `onboarding_paywall_cleared bool` to
  `user_profiles`; hydrate via `sync_all_user_data` / `gating_service.hydrateFromProfile`
  pattern so a reinstall/multi-device user isn't reset. (Latch especially must survive
  reinstall, else a paid user who reinstalls before RC restore could be re-walled — though
  the premium check also covers that case.)

## Test plan (0/11 paths currently covered — all new)

- Router `stageFor()` unit tests: all four branches (welcome / tour / paywall / app) +
  latch-set and premium-set both clear the wall.
- Tour resume: `advance()` persists index; relaunch resumes at saved index, not 0.
- Hard paywall widget: no X rendered in `onboardingHardGate`; Restore + legal present;
  trial-start sets latch + fires `trial_started`; offerings-fail shows Continue and does
  NOT set latch; premium user skips the wall entirely.
- Standalone `/paywall` regression: X + exit offer STILL present in `standaloneUpsell`.
- `trial_started` fires exactly once on success.

## Rollout (3 phases — matches prior plan's blast-radius sequencing)

1. **Migration + AppSession hydration** — add columns, hydrate `tourCompleted` /
   `paywallGateCleared` / premium into `AppSessionNotifier` (no behavior change yet).
2. **Tour-as-gate + resume + router stage** — behind `hard_paywall_after_tour_enabled=false`.
3. **Hard paywall mode + relocate paywall out of onboarding + instrumentation** — flip
   the flag on. Remove the in-onboarding paywall pages (22-26) once stable for 48h.

## Failure modes
| Codepath | Failure | Test? | Handled? | User sees |
|----------|---------|-------|----------|-----------|
| Router stage (async hydration race) | flags not loaded at first redirect | required | hydrate-before-router boot order | flicker / fail-open → CRITICAL if unhandled |
| Offerings load | StoreKit/network fail at no-X wall | required | temp let-in "Continue" | enters free, re-walled later |
| trial_started | event dropped | n/a | fire-and-forget | nothing (analytics only) |
| Tour resume | saved index out of range after tour edit | required | clamp to [0, len) | starts at 0 |
